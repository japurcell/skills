Short answer — follow SKILL.md: read the implementer's concerns and classify them. If they’re about correctness or scope, do NOT update tracking: clarify acceptance criteria/constraints and re-dispatch an implementer subagent (or escalate) to resolve the scope mismatch, verify tests/build, then update tracking only when a DONE (or harmless observation) is returned. If the concerns are mere observations, note them in the plan/todo/docs, record what was verified, then mark the task done in tracker. Always log the verification performed.
