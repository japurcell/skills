# Bootstrap / Override Synthesis

This document specifies the Phase 0 (Bootstrap) capability of the `coding-task-workflow` skill. Bootstrap inspects a target repository and generates or updates repo-local override files that all subsequent phases read. Those files must be committed before Phase 2 creates a worktree, or refreshed into an existing Phase 2 worktree, so later phases see the same overrides from the checkout they implement in.

---

## When to Run Bootstrap

Run bootstrap automatically when:
- `.coding-workflow/overrides/` does not exist.
- One or more required override files are missing.
- Override files contain entries with `source: inferred` that are more than 30 days old and material signals have changed (e.g., new CI workflows, updated `package.json`).
- The Phase 2 worktree exists but is missing `.coding-workflow/overrides/` or has an out-of-date override tree.

Run bootstrap on demand when:
- The user passes `BOOTSTRAP=only`.
- The user asks to "refresh" or "re-sync" the override files.
- The repo has undergone major refactoring since the last bootstrap.

---

## Signals to Inspect

Inspect these files in the target repository, in the order listed. Later signals override earlier ones for the same field.

### Tier 1 — Canonical docs (highest confidence)

| File | Fields inferred |
|------|----------------|
| `README.md` | project purpose, tech stack, quick-start commands |
| `AGENTS.md` / `COPILOT.md` | agent hints, conventions, known traps |
| `CONTRIBUTING.md` | branch naming, PR process, code-style rules |
| `docs/architecture.md` or `docs/ARCHITECTURE.md` | architecture overview |
| `docs/adr/` | architectural decision records |

### Tier 2 — Config-derived (high confidence)

| File | Fields inferred |
|------|----------------|
| `package.json` (scripts section) | test, lint, typecheck, build commands |
| `pyproject.toml` / `setup.cfg` | test commands, linting config |
| `go.mod` | module name, Go version, key dependencies |
| `Cargo.toml` | crate name, edition, key dependencies |
| `Makefile` | build and test targets |
| `.eslintrc*` / `.flake8` / `rustfmt.toml` / `golangci-lint.yml` | linting rules |
| `tsconfig.json` | TypeScript settings, path aliases |

### Tier 3 — CI-derived (high confidence for commands)

| File | Fields inferred |
|------|----------------|
| `.github/workflows/*.yml` | exact CI commands for test, lint, typecheck, build |
| `.travis.yml` | same |
| `.circleci/config.yml` | same |

### Tier 4 — Inferred (lower confidence)

| Signal | Fields inferred |
|--------|----------------|
| Lockfile presence (`package-lock.json`, `yarn.lock`, `poetry.lock`, `Cargo.lock`) | package manager |
| Directory structure (`src/`, `lib/`, `app/`, `pkg/`, `cmd/`) | project layout conventions |
| Test file naming (`*.test.ts`, `*_test.go`, `test_*.py`) | test framework and conventions |
| Import patterns in source files | key frameworks and versions |

---

## Inference Rules

For each override file, apply signals in tier order (Tier 1 → Tier 4). For each inferred value:
- Record `source`: one of `canonical-doc | config-derived | ci-derived | inferred`.
- Record `confidence`: `high | medium | low`.
- Record `evidence`: the file name and section that the value was derived from.
- If a value is already present in the override file with `source: manual`, do **not** overwrite it.
- Commit the resulting override files before Phase 2 worktree creation so the worktree inherits them as tracked files.

### Special rules

- **Prefer linking over duplicating**: if an existing doc already covers a topic (e.g., CONTRIBUTING.md defines the branch naming policy), write a reference link in the override file rather than copying the content.
- **Do not generate repo-policy rules that conflict with CONTRIBUTING.md**: if a policy is already stated in a canonical doc, the override file must reference that doc, not restate it.
- **Test commands from CI take precedence** over commands inferred from `package.json` scripts: CI commands are what actually runs in CI; local scripts may differ.

Records the tech stack, languages, frameworks, and key directory paths.

```yaml

# Generated by coding-task-workflow bootstrap phase

    evidence: tsconfig.json
```

## Bootstrap Commit Convention

Bootstrap changes are committed separately from work-item changes:

```
chore: bootstrap coding-workflow overrides

Generated or updated .coding-workflow/overrides/ based on:
- README.md
- package.json
- .github/workflows/ci.yml
- docs/architecture.md

Bootstrap run: 2026-04-23T15:00:00Z
```

---

## Handling Conflicts with Manual Edits

If an override file already exists and contains entries with `source: manual`:
1. Never overwrite `source: manual` entries.
2. Add new inferred entries below or in a separate section.
3. Note which entries were skipped because of manual overrides in the commit message.

---

## Bootstrap Limitations

- Cannot access private external documentation (Confluence, Notion, internal wikis).
- Cannot run the test suite to verify inferred test commands — mark confidence `medium` when only inferred, `high` when confirmed by CI.
- Cannot determine runtime environment variables required for tests; mark `agent-hints.md` with a note to add them manually.
- Shallow clones: if the repo is a shallow clone, import analysis may miss older patterns. Note this in `repo-inventory.yaml` under `meta.notes`.
